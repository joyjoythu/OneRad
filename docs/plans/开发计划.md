# AutoRadiomics Agent 开发计划

赛道B，20天，4人团队。目标：用户上传影像文件夹 + 临床表格 + 一句任务描述，系统自动完成影像组学分析，输出标准化 Word 报告。

团队：1 负责人 + 3 低年级同学。LLM：DeepSeek V4 API。

---

## 一、8 个 Agent 及职责

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Orchestrator** | 总控状态机，调度各 Agent，处理中断与恢复 | 用户请求 + 文件路径 | 逐阶段状态事件 |
| **Discovery Agent** | 扫描文件夹，将 image 与 mask 按患者 ID 配对 | 文件夹路径 | 配对列表 + 未配对文件列表 |
| **Clinical Agent** | 读取临床表格，调用 LLM 识别 ID 列、Label 列、特征列 | CSV/Excel 路径 | 列名分类结果（id_col, label_col, feature_cols） |
| **Matching Agent** | 将影像文件名中的患者 ID 与表格 ID 列对齐 | 影像 ID 列表 + 表格 ID 列 | 匹配成功的 DataFrame + 未匹配 ID 列表 |
| **QC Agent** | 数据质控：mask 非空检查、image/mask 尺寸一致性、spacing 检查与 resample、值域检查 | 配对列表 | 质检通过/失败结果 |
| **Feature Agent** | 调 PyRadiomics 提取影像组学特征，按 CT/MRI 自动选参数 | 配对列表 | 特征矩阵 DataFrame |
| **Analysis Agent** | 自动判断分析类型，LASSO 筛选 + Logistic Regression / CoxPH | 合并后的特征+临床+Label 数据 | 统计结果字典（系数、OR/HR、95%CI、p 值、AUC/C-index） |
| **Report Agent** | 将统计结果填入 Word 模板，生成学术报告 | 统计结果 + 原始数据信息 | Word 文档路径 |

**流水线：**

```
Discovery -> Clinical -> Matching -> QC -> Feature -> Merge -> Analysis -> Report
```

每个 Agent 失败 → Orchestrator 中断 → 用户选"跳过"或"终止" → 恢复或结束。

## 二、技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **LLM 编排** | LangChain LCEL（可选） | 只用 Runnable 和 PromptTemplate 管理 3 个 LLM 调用链，不碰 Agent/Memory/RAG |
| **LLM 模型** | DeepSeek V4 API | OpenAI 兼容格式，3 个调用点：意图解析、列名识别、报告润色 |
| **医学图像** | SimpleITK + PyRadiomics | 读图 + 特征提取，按模态自动选参数 |
| **统计分析** | scikit-learn（LASSO、LR）+ lifelines（CoxPH）+ scipy | 特征筛选 + 回归 + 统计检验 |
| **临床表格** | pandas + difflib | CSV/Excel 读写 + ID 模糊匹配 |
| **状态机** | 原生 Python（Enum + while） | Orchestrator 手写，不引入 LangGraph/CrewAI/AutoGen |
| **前端** | Gradio | SSE 流式进度 + 中断按钮 + 报告下载 |
| **报告** | python-docx | 直接生成 .docx |
| **部署** | Docker 单容器 | 一键启动 |

### 关于 LangChain 的判断

这条流水线确实像工作流，但 LLM 只参与 3 个孤立节点（意图解析、列名识别、报告润色），节点之间没有复杂的 Memory 共享或 Tool 调用。引入 LangChain 的**收益有限**：PromptTemplate 能统一管理 Prompt，LCEL 能把 LLM 调用包装成可复用的 Chain，调试起来稍微方便一点。但代价是低年级同学要多学一个框架，安装包也变大。

**建议：** 负责人用 LangChain 的 `PromptTemplate` + `Runnable` 封装 LLM 调用，写在一个文件里（`llm.py`），对外暴露普通函数接口。低年级同学无感知，只调函数。如果 Day 3 之前搭不起来，立刻回退到原生 OpenAI SDK，不硬撑。

### 砍掉的功能

以下功能时间不够，明确不做了：

- ICC 稳定性检查（以后有余力再加）
- 多模态融合（PET/CT 只做单个模态）
- React/Vue 精美前端
- Markdown 中间格式（直接出 Word）

## 三、模块划分与文件结构

```
app/
├── orchestrator.py      # Orchestrator 状态机（负责人）
├── llm.py               # DeepSeek API 封装，可选 LangChain LCEL（负责人）

└── ui.py                # Gradio 前端（同学 C）
```

## 四、4 人分工

| 角色 | 负责文件 | 任务边界 |
|------|----------|----------|
| **负责人** | `orchestrator.py`, `llm.py` | 写状态机骨架 + LLM 封装，定义各 Agent 输入输出接口（数据契约），Code Review + 联调。低年级同学不碰状态机和 LLM Prompt。 |
| **同学 A** | `discovery.py`, `qc.py` | Discovery Agent + QC Agent。输入文件夹路径，输出配对列表和质检结果。不碰 LLM、统计、前端。 |
| **同学 B** | `clinical.py`, `feature.py`, `analysis.py`, `report.py` | Clinical Agent + Matching Agent + Feature Agent + Analysis Agent + Report Agent。输入表格路径和影像特征矩阵，输出统计结果和 Word 报告。不碰文件扫描、前端、LLM Prompt。 |
| **同学 C** | `ui.py` | Gradio 前端。按负责人定义的 SSE 事件格式渲染界面（进度条、日志、中断按钮、报告下载）。不碰后端逻辑、影像处理、统计。 |

**铁律：** 每人只改自己的文件，跨文件调用走负责人定义的接口函数。

## 六、LLM 调用点（共 3 处）

| 调用点 | 输入 | 输出 | 用途 |
|--------|------|------|------|
| 意图解析 | 用户自然语言描述 | `{"task_type": "survival", "covariates": ["Age", "Sex"]}` | Analysis Agent 决定走哪条分支 |
| 列名识别 | 临床表格列名列表 | `{"id_col": "PatientID", "label_col": "OS", "feature_cols": [...]}` | Clinical Agent 自动认列 |
| 报告润色 | 原始统计结果字典 | 学术化描述段落 | Report Agent 填充方法学段落 |

## 七、Git 工作流

**分支命名：**
```
feat/discovery-agent
feat/clinical-matching
feat/word-report
fix/qc-mask-empty
docs/api-usage
```

**Commit 规范：**
```
feat: 添加文件自动配对算法
fix: 修复 mask 为空时的崩溃
docs: 更新 README 安装说明
test: 添加 Discovery Agent 单元测试
```

**铁律：**
- `main` 分支永远可运行，禁止直接 push
- 所有开发在 `feature/xxx` 分支进行
- 合并前必须通过 PR + 至少 1 人 Review
- 每天开工前 `git pull origin main`

## 九、总结

负责人手写状态机骨架，3 个同学各领封闭模块。LangChain 可选引入，但只用来管理 Prompt 和 LLM 调用链，不对低年级同学暴露。DeepSeek API 只用于 3 个孤立节点。影像特征用 PyRadiomics 提，临床表格用 pandas 桥接，Gradio 做前端，python-docx 出 Word 报告。20 天够跑通。
