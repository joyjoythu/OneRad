## 1. 设计目标与原则

### 1.1 领域背景

影像组学（Radiomics）是从医学影像中高通量提取定量特征并建立预测模型的技术。一次完整的分析涉及多个步骤：

```
图像/mask配对 → 临床表解析 → 参数确认 → QC → 特征提取 → 特征筛选+建模 → 报告生成
```

传统做法需要研究者手动切换多个 Python 脚本，每一步都涉及参数判断。OneRad Agent 的目标是**让临床研究者用自然语言描述分析意图**，Agent 在理解项目上下文后自主规划并执行流程，同时在关键操作节点（文件修改、脚本执行、耗时计算）征得用户确认。

### 1.2 核心设计原则

| 原则 | 说明 | 在代码中的体现 |
|------|------|---------------|
| **安全可控** | 所有副作用操作均需用户确认；路径活动被沙箱限制 | `Sandbox.resolve()` 拒绝越界路径；`process_tool_calls` 中 `confirmation_pending` 阻止静默执行 |
| **人机协同** | 采用 LangGraph `interrupt()` 挂起-确认-恢复循环，而非事后通知 | `human_review` 节点调用 `interrupt()` 序列化中断状态；前端渲染对应审批面板；用户操作通过 `Command(resume=...)` 恢复 |
| **可观测** | 思考链、提取进度、子Agent 状态通过 SSE 实时推送 | `_publish_thinking` / `_publish_agent_progress` / `_publish_subagent` 三个旁路发布函数 |
| **上下文高效** | 子Agent 在隔离上下文中运行，只有结论回到主对话 | `_run_subagent` 独立图 + 独立 MemorySaver；`_SUBAGENT_RESULT_MAX_CHARS = 4000` |
| **热更新友好** | System prompt 每次从磁盘读取，编辑即生效 | `skills.py` 的 `load_skill()` 无缓存设计 |

## 3. LangGraph 状态机详解

### 3.1 图结构（5 节点 + 2 条件边）

```python
# app/agent/graph.py
def create_agent_graph(checkpointer=None):
    builder = StateGraph(AgentState)

    builder.add_node("call_llm", call_llm)             # 节点1: LLM推理
    builder.add_node("process_tool_calls", process_tool_calls)  # 节点2: 解析工具调用
    builder.add_node("human_review", human_review)      # 节点3: 人工审批
    builder.add_node("auto_confirm", auto_confirm)      # 节点4: 自动审批
    builder.add_node("execute_confirmed", execute_confirmed) # 节点5: 执行确认

    builder.add_edge(START, "call_llm")
    builder.add_conditional_edges("call_llm", should_continue, {
        "process_tool_calls": "process_tool_calls",
        "__end__": END
    })
    builder.add_conditional_edges("process_tool_calls", route_after_process, {
        "human_review": "human_review",
        "auto_confirm": "auto_confirm",
        "call_llm": "call_llm"
    })
    builder.add_edge("human_review", "execute_confirmed")
    builder.add_edge("auto_confirm", "execute_confirmed")
    builder.add_edge("execute_confirmed", "call_llm")

    return builder.compile(checkpointer=checkpointer)
```

### 3.2 完整流转路径

一次完整的工具调用生命周期经过以下 superstep：

```
Superstep 1: call_llm
  ├── 组装 SystemMessage (skills/ 加载) + 历史 messages
  ├── 注册 17 个工具（根据 allow_subagent/readonly 动态裁剪）
  ├── 流式调用 DeepSeek
  │   ├── 每个 delta chunk 的 reasoning_content → _publish_thinking (SSE旁路)
  │   └── 累积 content + tool_calls → 组装 AIMessage
  └── 输出: AIMessage (content + tool_calls + reasoning_content 挂 additional_kwargs)

Superstep 2: process_tool_calls (if has tool_calls)
  ├── 遍历每个 tool_call
  ├── 按工具名路由：
  │   ├── update_todo_list → 直接写入 state.todos，返回 ToolMessage
  │   ├── dispatch_subagent(explore) → 直接并行执行子Agent，返回 ToolMessage
  │   ├── 需确认工具 → 设置 pending_* + interrupt_type
  │   └── 免确认工具 → 直接执行，返回 ToolMessage
  └── 约束：同一轮至多一个 pending_* 非空（confirmation_pending 标志）

Superstep 3a: auto_confirm (if auto_approve=True)
  └── 直接输出 {"confirmed": True}

Superstep 3b: human_review (if auto_approve=False)
  └── interrupt({"type": ..., pending_*: ...})
      └── 图挂起，序列化中断状态到 checkpoint
      └── 等待外部 Command(resume={"action": "confirm"|"cancel"|"other"})

Superstep 4: execute_confirmed
  ├── 根据 interrupt_type 取对应的 pending_* 字段
  ├── if confirmed:
  │   ├── file_plan → execute_plan() 执行文件操作
  │   ├── system_command → Sandbox 限制下执行命令
  │   ├── python_script → execute_script_if_safe()（low直接运行/medium注入沙箱头）
  │   ├── radiomics_execution → FeatureAgent.run()（在独立线程中提特征）
  │   ├── radiomics_analysis → run_radiomics_cv_analysis()
  │   ├── feature_statistics → run_feature_statistics()
  │   └── subagent_dispatch → _run_subagents()（并行子Agent）
  │   └── 结果注入 {"executed": True, "note": "操作已执行完成..."}
  ├── else:
  │   └── {"cancelled": True} + 可选 HumanMessage
  └── _clear_interrupt() → 清空所有 pending_* 字段

Superstep 5: call_llm (循环回到步骤1)
```

### 3.3 Checkpoint 持久化

使用 LangGraph 的 `AsyncSqliteSaver`，每次 superstep 结束后自动将 `AgentState` 完整快照写入 SQLite。关键行为：

- **持久化内容**：`messages` 列表（含 `additional_kwargs.reasoning_content`）、`todos`、`model`、`project_path`、`base_url`
- **不持久化**：`api_key`（走 `RunnableConfig.configurable`）
- **interrupt 恢复**：`Command(resume=...)` 触发时，LangGraph 从 checkpoint 恢复状态并从中断点继续
- **容错**：`/stop` 后清理残留的 `interrupt_type` 和 `pending_*` 字段，防止前端误弹审批面板。若消息历史末尾有未应答的 `tool_calls`（缺少对应 `ToolMessage`），为每个缺失 id 补一条 `{"cancelled": true}` 的 ToolMessage，避免下次 LLM 调用 400

## 4. 状态设计

### 4.1 AgentState 完整字段语义

```python
# app/agent/state.py
class AgentState(TypedDict):
    # ═══════════════════════════════════════════════════════════
    # 对话核心
    # ═══════════════════════════════════════════════════════════
    messages: Annotated[list, add_messages]
    # LangChain 消息累积器。add_messages reducer 保证新消息追加而非覆盖。
    # 包含 HumanMessage / AIMessage / ToolMessage / SystemMessage 四种类型。
    # AIMessage.additional_kwargs 携带 reasoning_content（思考链全文）和
    # timestamp（UTC ISO 8601）。ToolMessage 的 tool_call_id 与 AIMessage
    # tool_calls[*].id 一一对应。

    project_path: str     # 当前项目根目录（绝对路径）
    base_url: str         # LLM API 地址，固定为 "https://api.deepseek.com/v1"
    model: str            # 当前会话选用的模型名。新消息可携带 model 覆盖此值

    # ═══════════════════════════════════════════════════════════
    # 安全凭证（设计上不持久化到 checkpoint）
    # ═══════════════════════════════════════════════════════════
    api_key: NotRequired[Optional[str]]
    # 仅兼容旧版 checkpoint。新线程通过 RunnableConfig.configurable.api_key
    # 传入，避免密钥写入 SQLite。

    # ═══════════════════════════════════════════════════════════
    # 中断状态——同一时刻至多一个非 None
    # ═══════════════════════════════════════════════════════════
    interrupt_type: Optional[str]
    # 8 种值之一：
    #   "file_plan"           — plan_file_operations
    #   "system_command"      — list_directory / find_files / get_file_info /
    #                           read_yaml / read_tabular_file / update_yaml /
    #                           inspect_image_spacing
    #   "python_script"       — execute_python_script
    #   "radiomics_plan"      — discover_radiomics_pairs
    #   "radiomics_execution" — extract_radiomics_features
    #   "radiomics_analysis"  — run_radiomics_analysis
    #   "feature_statistics"  — run_feature_statistics
    #   "subagent_dispatch"   — dispatch_subagent (general 模式)
    # 额外的非中断交互：
    #   "choice"              — ask_user_choice（通过 pending_choice 字段承载，
    #                            不经过 human_review，前端选择面板独立处理）

    pending_plan: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "plan": [{"action": "move"|"copy"|"rename"|"mkdir",
    #         "source": str, "target": str, "reason": str, "overwrite": bool}]}
    # plan 已由 validate_plan() 校验，所有路径已通过沙箱检查。

    pending_command: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "_pending_tool": str, "args": {...}}
    # 前端 CommandPanel 按 _pending_tool 决定展示格式。

    pending_script: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "code": str, "description": str,
    #         "risk_level": "low"|"medium"|"high", "script_path": str}

    pending_radiomics_plan: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "images_found": int, "masks_found": int,
    #         "pairs": {"high": [...], "medium": [...], "low": [...]},
    #         "unmatched_images": [...], "unmatched_masks": [...]}

    pending_radiomics_execution: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "pairs": [...], "n_cases": int,
    #         "yaml_path": str, "output_dir": str, "expected_outputs": [...]}

    pending_radiomics_analysis: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "feature_csv": str, "clinical": str,
    #         "id_col": str, "label_col": str, "covariates": [...],
    #         "output_dir": str, "n_feature_cases": int, "n_matched": int, ...}

    pending_feature_statistics: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "feature_csv": str, "clinical": str,
    #         "selected_features_csv": str, "selected_features": [...], ...}

    pending_subagent: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "tasks": [str, ...], "mode": "general"}

    pending_choice: Optional[Dict[str, Any]]
    # 结构: {"tool_call_id": str, "question": str, "options": [str, ...]}

    # ═══════════════════════════════════════════════════════════
    # 用户反馈
    # ═══════════════════════════════════════════════════════════
    confirmed: Optional[bool]            # True=确认 / False=取消
    choice_answer: Optional[str]         # 用户在选择面板提交的答案
    other_instruction: Optional[str]     # 用户选择"其他"时输入的替代指令

    # ═══════════════════════════════════════════════════════════
    # UI 辅助字段
    # ═══════════════════════════════════════════════════════════
    context_usage: Optional[Dict[str, Any]]
    # {"input_tokens": int, "output_tokens": int, "total_tokens": int}
    # 最近一次 LLM 调用的 token 用量（从 AIMessage.usage_metadata 提取）

    todos: NotRequired[List[Dict[str, str]]]
    # [{"content": "步骤描述", "status": "pending"|"in_progress"|"completed"}]
    # 由 update_todo_list 工具全量替换，前端 TodoPanel 渲染

    tool_outputs: Annotated[list, lambda x, y: (x or []) + y]
    operation_log: Annotated[list, lambda x, y: (x or []) + y]
    # 累积型列表字段。operation_log 用于前端状态栏，tool_outputs 用于日志面板
```

### 4.2 关键约束：同一轮只允许一个需确认的工具调用

LLM 可能在单次 `call_llm` 响应中并行发出多个 `tool_calls`（例如同时 `list_directory` 和 `read_yaml`）。虽然在 API 层面已设置 `parallel_tool_calls=False`，但工具定义层面仍需防御。

`process_tool_calls` 中的处理逻辑：

```
for each tool_call:
    if name in 需确认集合:
        if confirmation_pending:
            → 返回 error ToolMessage（保证每个 tool_call_id 都有响应）
            → continue
        confirmation_pending = True
        设置对应的 pending_* + interrupt_type
```

**设计理由**：如果允许多个 pending 状态，第二个会覆盖第一个的 `pending_*` 字段，导致第一个的 `tool_call_id` 永远收不到 ToolMessage，下次 LLM 调用时历史中 assistant 的 tool_calls 缺少对应 tool 消息，API 返回 400。

### 4.3 API Key 的不落库策略

```python
# _resolve_api_key 的三级优先级：
# 1. RunnableConfig.configurable.api_key  （运行时注入，不持久化）
# 2. state.api_key                          （兼容旧 checkpoint）
# 3. 环境变量 OPENAI_API_KEY / DEEPSEEK_API_KEY
```

创建和恢复线程时，前端传回的 API Key 存储在 `app.state.agent_api_keys[thread_id]`（FastAPI 内存字典），在 `_agent_config()` 中注入 `RunnableConfig`。线程删除时同步清理。

## 5. 工具体系

### 5.1 完整工具目录

#### 5.1.1 只读探索工具

| 工具 | 参数 | 返回值 | 确认 | readonly |
|------|------|--------|------|----------|
| `list_directory` | `path: str` | `{"_pending_tool": "list_directory", "args": {"path": ...}}` | 需要 | ✅ |
| `find_files` | `pattern: str`, `path: str = "."` | `{"_pending_tool": "find_files", "args": {...}}` | 需要 | ✅ |
| `get_file_info` | `path: str` | `{"_pending_tool": "get_file_info", "args": {...}}` | 需要 | ✅ |
| `read_yaml` | `path: str`, `key: str = ""` | `{"_pending_tool": "read_yaml", "args": {...}}` | 需要 | ✅ |
| `read_tabular_file` | `path: str`, `sheet_name: str = ""`, `head: int = 20`, `columns: List[str] = None` | `{"_pending_tool": "read_tabular_file", "args": {...}}` | 需要 | ✅ |
| `discover_radiomics_pairs` | 无 | `{"_pending_tool": "discover_radiomics_pairs", ...}` | 需要 | ✅ |
| `inspect_image_spacing` | `pairs: List[Dict] = None` | `{"_pending_tool": "inspect_image_spacing", "args": {...}}` | 需要 | ✅ |

**read_tabular_file 的智能预处理**：自动尝试 UTF-8/GBK 编码；`columns` 可只读指定列；`head=0` 时只返回结构不取数据；结果 JSON 限制在 ~12KB 以内（超宽表时逐半减半 head_rows）。

**discover_radiomics_pairs 的配对算法**：递归扫描 `images/` 和 `masks/` 目录的 `.nii.gz` 文件 → 按文件名后缀匹配（`_mask`/`_seg`/`_label`/`_roi`）→ 按 token 交集匹配 → 按患者 ID 前缀匹配 → 输出高/中/低置信度三级配对结果。

#### 5.1.2 写入配置工具

| 工具 | 参数 | 说明 |
|------|------|------|
| `update_yaml` | `path: str`, `updates: Dict[str, Any]` | 点号路径到值的映射，如 `{"setting.binWidth": 10}`。保留注释与格式（`ruamel.yaml`），中间层级自动创建 |

#### 5.1.3 计划与执行工具

| 工具 | 参数 | 说明 |
|------|------|------|
| `plan_file_operations` | `instruction: str` | 调用 LLM（独立实例）生成文件操作计划 → `validate_plan()` 校验。只允许 move/copy/rename/mkdir 四种操作。实际执行在确认后走 `execute_plan()`（含自动备份到 `.onerad_backup/`） |
| `execute_python_script` | `description: str`, `code: str` | 保存脚本到 `agent_scripts/` 目录 → AST 风险分级 → 确认后执行。low 直接运行，medium 注入沙箱头后运行，high 经用户知情确认后运行 |

**Python 脚本风险分级的 AST 分析**（`classify_risk()`）：

```
高危判定 (high):
  - import socket / subprocess / ctypes / urllib / requests / httpx / ...
  - from os import system / from shutil import rmtree
  - 调用 eval() / exec() / os.system() / subprocess.Popen()
  - 字符串中出现 __builtins__ / __import__

中危判定 (medium):
  - 导入了既非高危也非白名单的模块（如 numpy, pandas, nibabel）
  - 写文件操作（open(..., 'w')）
  - 路径遍历模式（'../'）
  - 绝对路径引用
  - 调用 remove() / unlink() / getattr() / Process()

低危 (low):
  - 仅使用白名单模块（math, json, csv, re, collections, ...）
  - 无写操作、无网络、无子进程
```

**沙箱头注入**（仅 medium 脚本）：在脚本开头注入 Hook 代码，运行时替换 `builtins.open` 和 `io.open`，对越界访问抛出 `PermissionError`。注意这是 best-effort 防御，不是安全边界（无法防护系统调用和直接使用 `os` 模块）。

### 6.3 取消的语义处理

用户取消时，`execute_confirmed` 的行为因操作类型而异：

```
if 简单查询取消 (system_command):
    → 仅返回 cancelled ToolMessage
    → LLM 自然知道"操作没做"，不会误判

if 重操作取消 (file_plan / python_script / radiomics_* /
              feature_statistics / subagent_dispatch):
    → 返回 cancelled ToolMessage
    → 追加 HumanMessage("我取消了刚才的操作，请不要重试。请询问我现在想做什么。")
    → 防止 LLM 将"取消结果"误读为"操作失败"而自动重试
```

## 7. 子Agent系统

### 7.1 设计动机与核心思想

完整的影像组学分析需要在项目目录中探索大量信息：目录结构、文件清单、YAML 配置、临床表格列名、图像/mask 配对情况。如果串行在主对话中完成，每一步的中间结果都累积在消息历史中，迅速消耗上下文窗口。

子Agent系统的核心思想：
1. **并行加速**：多个独立探索任务同时运行
2. **上下文隔离**：子Agent 的中间过程不进主对话，只有结论返回
3. **安全分级**：explore 模式只读，general 模式需确认

### 7.2 两种模式的完整对比

| 维度 | `mode="explore"` | `mode="general"` |
|------|-----------------|-----------------|
| **工具集** | 9 个（只读探索 + ask_user_choice） | 15 个（全部除 dispatch_subagent） |
| **confirm 流程** | 无——在 process_tool_calls 中直接执行 | 有——走 auto_confirm（子Agent 内部 auto_approve=True） |
| **可见性** | 中间过程只推前端，不进主对话 | 中间过程只推前端，不进主对话 |
| **典型用法** | 项目开局探索：扫目录/读配置/查临床表 | 需要写操作或计算的子任务 |
| **并行上限** | 4 worker | 4 worker |
| **LLM 调用** | 每次工具调用自动批准 | 每次工具调用自动批准（但 high 脚本危险标记仍保留） |

### 7.3 隔离机制的实现细节

```python
# _run_subagent 的核心流程

def _run_subagent(task, state, config, parent_thread_id, mode):
    # 1. 派生 thread_id
    sub_thread_id = f"{parent_thread_id}:sub:{uuid4().hex[:8]}"

    # 2. 注册运行时上下文（共享父线程取消事件）
    parent_ctx = agent_runtime.get(parent_thread_id)
    sub_ctx = agent_runtime.register(sub_thread_id)
    if parent_ctx is not None:
        sub_ctx.cancel_event = parent_ctx.cancel_event  # 取消传播

    # 3. 构建独立图 + 独立 MemorySaver
    sub_graph = create_agent_graph()  # 默认 MemorySaver，不共享父 checkpoint

    # 4. 组装子配置
    sub_config = {
        "configurable": {
            "thread_id": sub_thread_id,
            "auto_approve": True,        # 免确认，不会 interrupt
            "allow_subagent": False,     # 禁止再嵌套分派
            "readonly_tools": mode == "explore",
            "api_key": ...,
        },
        "recursion_limit": 150,          # ~37 轮工具调用
    }

    # 5. 注入专用 System Prompt
    system_prompt = (
        EXPLORE_SUBAGENT_SYSTEM_PROMPT  # "你是只读探索子 agent..."
        if mode == "explore"
        else SUBAGENT_SYSTEM_PROMPT      # "你是全功能子 agent..."
    )

    # 6. 流式运行
    sub_input = {
        "messages": [SystemMessage(content=system_prompt),
                     HumanMessage(content=task)],
        "project_path": state["project_path"],
        ...
    }

    for values in sub_graph.stream(sub_input, sub_config, stream_mode="values"):
        # 滚动推送中间过程到前端
        status["entries"] = _summarize_subagent_entries(values["messages"])
        _publish_subagent(parent_thread_id, status, persist=False)

        # 检查取消
        if sub_ctx.cancel_event.is_set():
            cancelled = True
            break

    # 7. 提取结论
    conclusion, usage = _extract_subagent_result(final["messages"])
    # 结论 = 最后一条非空 AIMessage.content
    # usage = 所有消息 usage_metadata 的累加

    # 8. 截断返回
    return {
        "success": True,
        "result": conclusion[:_SUBAGENT_RESULT_MAX_CHARS],  # 4000 字符
        "usage": usage,
    }
```

### 7.4 上下文截断策略

```
子Agent 消息历史:
  [SystemMessage, HumanMessage(task),
   AIMessage(tool_calls), ToolMessage, AIMessage(content),
   AIMessage(tool_calls), ToolMessage, AIMessage(content),
   ...中间过程...,
   AIMessage(最终结论)]

进入主对话的内容:
  → 最终结论截断为 4000 字符 (_SUBAGENT_RESULT_MAX_CHARS)
  → 累计 token 用量汇总到 dispatch_subagent 返回的 usage 字段

推送到前端的内容:
  → 滚动窗口：最近 8 条消息摘要 (_SUBAGENT_ENTRY_WINDOW)
  → 每条摘要截断为 300/200 字符
  → 中间滚动 persist=False（不写 SQLite），开始/结束 persist=True
```

### 7.5 并行执行

```python
# _run_subagents 中的并发控制

if len(tasks) == 1:
    results = [_run_subagent(tasks[0], ...)]          # 单任务：内联执行
else:
    workers = min(len(tasks), _SUBAGENT_MAX_WORKERS)  # 最多 4 并发
    with ThreadPoolExecutor(max_workers=workers) as pool:
        futures = [pool.submit(_run_subagent, task, ...) for task in tasks]
        results = [f.result() for f in futures]        # 阻塞等待全部完成
```

每个子Agent 在独立线程中运行，各自持有独立的图实例和 MemorySaver。线程安全由各模块的独立实例保证（无共享状态）。

  → parent_ctx.cancel_event.set()

子Agent 中检测：
  → 每次 stream_mode="values" 迭代后检查 sub_ctx.cancel_event.is_set()

  → 检测到取消 → 立即退出 → status="cancelled" → 返回 {"success": False, "cancelled": True}

## 8. 流式通信协议

### 8.1 为什么绕过 LangChain 的 ChatOpenAI

LangChain 的 `ChatOpenAI` 在流式模式下调用 `openai.ChatCompletion.create(stream=True)`，内部用 `ChatCompletionChunk` 拼接最终 `AIMessage`。DeepSeek 在标准 OpenAI 协议之外增加了 `reasoning_content` 字段（思考链），该字段出现在 delta chunk 中但**不在**最终的 `ChatCompletion` 响应里。LangChain 的拼接逻辑不认识这个字段，直接丢弃。

解决方案：在 `call_llm` 中直接用**原生 OpenAI SDK** 做流式调用：

```python
# _stream_chat_completion 的关键逻辑

stream = client.chat.completions.create(
    model=model,
    messages=convert_to_openai_messages(messages),
    tools=[convert_to_openai_tool(t) for t in tools],
    temperature=0.2,
    parallel_tool_calls=False,
    stream=True,
    stream_options={"include_usage": True},
)

reasoning_parts = []       # 思考链累积
content_parts = []          # 文本累积
tool_slots = {}             # 按 index 分槽累积 tool_calls

for chunk in stream:
    # usage chunk（开启 include_usage 后流末尾会有一个仅含 usage 的 chunk）
    if chunk.usage:
        usage_metadata = {...}

    delta = chunk.choices[0].delta

    # reasoning_content：非标准字段，每 chunk 带增量或全量
    if delta.reasoning_content:
        reasoning_parts.append(delta.reasoning_content)
        _publish_thinking(thread_id, "".join(reasoning_parts), done=False)

    # content：标准字段
    if delta.content:
        content_parts.append(delta.content)

    # tool_calls：按 index 分槽累积（OpenAI 流式中同一个 tool_call 跨多个 chunk 发送）
    for tc in (delta.tool_calls or []):
        slot = tool_slots.setdefault(tc.index, {"id": "", "name": "", "arguments": ""})
        if tc.id: slot["id"] += tc.id
        if tc.function:
            if tc.function.name: slot["name"] += tc.function.name
            if tc.function.arguments: slot["arguments"] += tc.function.arguments

# 流结束
_publish_thinking(thread_id, full_reasoning, done=True)

# 组装 AIMessage
return AIMessage(
    content="".join(content_parts),
    tool_calls=[...],  # 按 index 排序，arguments JSON.parse
    additional_kwargs={"reasoning_content": full_reasoning},  # 随 checkpoint 持久化
    usage_metadata=usage_metadata,
)
```

### 8.2 SSE 事件体系

#### 8.2.1 事件类型一览

| 事件来源 | SSE event 类型 | persist | 触发时机 | 数据格式 |
|---------|---------------|---------|---------|---------|
| graph values 快照 | `agent` | ✅ | 每个 superstep 结束 | `AgentState`（messages + interrupt_type + pending_* + todos + context_usage + running） |
| 思考链 | `agent` | ❌ | 流式 chunk 到达 | `{"thinking": {"text": "...", "done": false}, "running": true}` |
| 提取进度 | `agent` | ✅ | 每个病例提完 | `{"radiomics_progress": {"stage": "extracting", "current": 5, "total": 30, "patient_id": "..."}, "running": true}` |
| 子Agent 状态 | `agent` | 开始/结束 ✅，中间滚动 ❌ | 子Agent 状态变化 | `{"subagent": {"id": "...", "task": "...", "status": "running", "entries": [...]}, "running": true}` |
| 流结束 | `agent_end` | — | async generator 正常结束 | `{}` |
| 心跳 | `: keep-alive` | — | 每秒（无新事件时） | 注释行 |

#### 8.2.2 persist 策略的设计逻辑

```
高频事件 (persist=False) — 每秒可能推送多次:
  - thinking delta: 推理过程中每个 token chunk 都推送累积全文
  - subagent 中间滚动: 每个 tool call 轮次都更新

原因:
  - SQLite 写入成为瓶颈（每秒数十次 INSERT）
  - 回放无价值（用户只需要看到最新状态）

兜底:
  - 页面刷新后调用 GET /threads/{id} 获取完整 values 快照
  - 快照中 AIMessage.additional_kwargs.reasoning_content 含完整思考链
  - 子Agent 开始/结束状态 persist=True（重连后可判断是否有正在运行的子Agent）

低频事件 (persist=True):
  - graph values 快照: 每个 superstep 一次
  - 提取进度: 每个病例一次（典型场景 30-100 个病例）
  - 子Agent 开始/结束
```

#### 8.2.3 进度补偿机制

```
场景：用户刷新页面时，特征提取正在进行中

正常流程（无刷新）:
  extract case 1 → publish(radiomics_progress, persist=True)
  extract case 2 → publish(radiomics_progress, persist=True)
  ...

刷新后:
  GET /threads/{id} → values 快照（不含 radiomics_progress，因为不在 state 中）
  连接 SSE → 只接收新事件（订阅时的 last_event_id = 最新持久化事件 id）

问题：刷新前的进度事件不会重放，前端进度条消失。

解决方案（在 thread_events 端点中）:
  if fresh_subscribe and thread_id in active_agent_streams:
      latest = bridge.store.get_latest_sse_event_containing(
          "agent", thread_id, "radiomics_progress"
      )
      if latest:
          catch_up_data = latest["data"]  # 最近一条含 radiomics_progress 的持久化事件

  在 event_generator 中:
      if catch_up_data:
          yield f"event: agent\ndata: {catch_up_data}\n\n"  # 不带 id 行！
          # ↑ 不带 id 是为了让浏览器的 Last-Event-ID 保持未设置，
          # 自动重连时走同样的补偿逻辑而非回放全部历史
```

### 8.3 EventBridge 设计

```python
# app/api/sse.py - EventBridge 的核心数据结构

class EventBridge:
    store: ProjectStore          # SQLite 持久化
    _queues: Dict[str, Dict[int, asyncio.Queue]]
    # "agent:{thread_id}" → {id(queue): asyncio.Queue(maxsize=256)}
    _locks: Dict[str, asyncio.Lock]    # 每 scope 一把锁
    _next_ids: Dict[str, int]           # 每 scope 的单调 id 分配器
    _pending_writes: Dict[str, Set[asyncio.Task]]  # 每 scope 的进行中写任务
```

**publish 的原子性保证**：

```python
async def publish(scope, scope_id, data, *, persist=True):
    async with lock:                          # scope 锁
        event_id = allocate_event_id()        # 单调递增分配

        if persist:
            write_task = asyncio.ensure_future(
                run_in_threadpool(store.record_sse_event, ...)
            )
            track_write(write_task)
            await asyncio.shield(write_task)  # 写任务不被发布者取消传播

        for queue in subscribers.values():    # 推送到在线订阅者
            if queue.full():
                queue.get_nowait()            # 丢弃最旧事件
            await queue.put({"event_id": event_id, "data": data})

    return event_id
```

**subscribe 的间隙消除**：

```python
async def subscribe(scope, scope_id, last_event_id):
    async with lock:
        await await_pending_writes()          # 等所有进行中写任务落库
        max_event_id = next_event_id() - 1    # 确认数据库最大 id

        register_queue()                      # 注册新订阅者
        historical = store.list_sse_events(   # 回放历史事件
            after_event_id=last_event_id)
        for event in historical:
            if event["event_id"] <= max_event_id:
                await queue.put(event)        # 先放历史

    return queue                              # 之后放实时的
    # 在锁内完成回放+注册，保证不遗漏间隙事件
```

### 8.4 前端 SSE 消费

```typescript
// frontend/src/api/agent.ts - connectAgentEvents

const es = new EventSource(`${API_BASE}/agent/threads/${threadId}/events`)

es.addEventListener('agent', (event: MessageEvent) => {
    const data = JSON.parse(event.data) as AgentState
    callbacks.onState?.(data)  // → applyState() → 更新 Pinia store
})

es.addEventListener('agent_end', () => {
    callbacks.onEnd?.()
    // → busy = false, radiomicsProgress = null, currentThinking = null
    // → syncThread() 获取最终 state 保证收敛
    // → listThreads() 刷新侧边栏（首轮对话后标题已生成）
})

es.onerror = () => {
    callbacks.onError?.({ message: 'EventSource error' })
    // → currentThinking = null (陈旧 thinking 无快照兜底)
    // → disconnect()
    // 浏览器会自动重连（EventSource 标准行为），重连时携带 Last-Event-ID
}
```

**前端状态更新的关键约束**：`applyState()` 中 `busy` 完全由后端上报的 `running` 字段驱动，**不能**由 `interrupt_type` 推断。原因：`execute_confirmed` 清除 `interrupt_type` 前会有一个中间快照仍带旧值——如果用 `interrupt_type != null` 判断 busy，确认后会短暂解除 busy 再恢复，导致 UI 闪烁。

## 9. 安全设计

### 9.1 路径沙箱

```python
# app/agent/safety.py

class Sandbox:
    def __init__(self, root: str):
        self.root = Path(root).resolve()      # 必须是已存在目录

    def resolve(self, path, must_exist=False):
        target = (self.root / path).resolve()  # 相对路径从 root 起算
        target.relative_to(self.root)          # 越界抛 ValueError
        if must_exist and not target.exists():
            raise FileNotFoundError(...)
        return target
```

所有工具函数在访问文件系统前**必须**通过 `sandbox.resolve()`。`execute_confirmed` 中有专门的 `PathEscapeError` 捕获（转义错误 → 返回 `{"success": False, "error": "路径超出项目目录"}`）。

### 9.2 确认-执行分离

```
传统 Agent 工具设计:
  LLM 调用 → 工具函数直接执行 → 返回结果

OneRad 工具设计:
  LLM 调用 → 工具函数校验参数 → 返回 {"_pending_tool": ..., 参数}
           → process_tool_calls 识别为需确认
           → human_review / auto_confirm
           → execute_confirmed 实际执行
```

这种两层设计的优势：
1. **LLM 和用户在审批面板看到相同的参数**——由工具函数校验后生成，而非 LLM 直接拼 JSON
2. **用户可在审批前编辑**——如修改文件操作计划的路径、增删条目
3. **审批面板可以格式化展示**——代码高亮、配对置信度表格、参数列表

### 9.3 纵深防御链

```
第一层: System Prompt (agent-core SKILL.md)
  "Respect every confirmation step, sandbox boundary, schema constraint,
   and risk decision enforced by the application."

第二层: 工具函数参数校验
  - extract_radiomics_features: 校验 YAML 存在 + 路径在沙箱内 + YAML 可解析
  - plan_file_operations: validate_plan() 限制四种操作 + 沙箱检查
  - execute_python_script: classify_risk() AST 静态扫描

第三层: process_tool_calls 确认拦截
  - 所有副作用工具必须通过 human_review/auto_confirm

第四层: execute_confirmed 运行时沙箱
  - Sandbox.resolve() 路径解析
  - 中危脚本注入沙箱头（运行时 hook open()）

第五层: 操作系统
  - Docker 容器 + 只读挂载（生产部署时）
```

## 11. 运行时上下文与取消机制

### 11.1 跨线程通信问题

LangGraph 的图节点（`call_llm`, `execute_confirmed` 等）是普通的同步函数，运行在 `run_in_threadpool` 的工作线程中。它们**无法直接访问** FastAPI 的 `app.state` 或 `asyncio` 事件循环。

而耗时操作需要：
1. **推送实时进度** → 需要访问 `EventBridge` 和主事件循环
2. **响应 /stop** → 需要一个线程安全的中断信号

### 11.2 AgentRunContext 注册表

```python
# app/agent/runtime.py

@dataclass
class AgentRunContext:
    cancel_event: threading.Event  # /stop 置位，耗时任务轮询
    loop: Any = None               # 主事件循环引用（用于 run_coroutine_threadsafe）
    bridge: Any = None             # EventBridge 实例

_lock = threading.Lock()
_contexts: Dict[str, AgentRunContext] = {}

def register(thread_id, loop=None, bridge=None):
    ctx = AgentRunContext(cancel_event=threading.Event(), loop=loop, bridge=bridge)
    with _lock:
        _contexts[thread_id] = ctx
    return ctx

def get(thread_id):
    with _lock:
        return _contexts.get(thread_id)

def request_cancel(thread_id):
    ctx = get(thread_id)
    if ctx is None: return False
    ctx.cancel_event.set()
    return True
```

### 11.3 进度推送的跨线程路径

```
工作线程 (execute_confirmed)                  主线程 (asyncio event loop)
        │                                              │
        ▼                                              │
_publish_agent_progress(thread_id, payload)            │
        │                                              │
        ▼                                              │
agent_runtime.get(thread_id)                           │
        │                                              │
        ├── ctx is None → 静默跳过                      │
        ├── ctx.cancel_event.is_set() → 跳过            │
        │                                              │
        ▼                                              │
asyncio.run_coroutine_threadsafe(                      │
    ctx.bridge.publish("agent", thread_id,             │
        {"radiomics_progress": payload,                │
         "running": True}),                            │
    ctx.loop                                           │
) ─────────────────────────────────────────────────▶  bridge.publish()
        │                                              │
        ▼                                              │
    返回继续提取下一个病例                               │
```

**关键保护**：`cancel_event.is_set()` 检查在 publish 之前。因为 `/stop` → `cancel_event.set()` → async task cancel 之间有时间窗口，worker 线程可能仍在推送进度——若推送 `running: True` 会在前端重新显示 busy 状态。

### 11.4 取消的协作式实现

```
/stop 端点:
  1. request_cancel(thread_id) → cancel_event.set()
     （必须优先执行：后续检查可能因 409 拒绝，但取消信号必须送达）

  2. task.cancel() → asyncio task 取消
     （取消 async generator → finally 清理 → 推送 running=false）

工作线程中的轮询:
  FeatureAgent.run(pairs, cancel_event=cancel_event, ...):
      for case in pairs:
          extract_one(case)
          if cancel_event.is_set():      # ← 协作式检查点
              return {"success": False, "cancelled": True}
          progress_callback(...)

子Agent 中的轮询:
  for values in sub_graph.stream(...):
      if cancel_event.is_set():
          cancelled = True
          break
```

**注意**：Docker/Windows 均无法安全地强杀线程。`cancel_event` 是协作式的——耗时任务必须在合适的检查点主动轮询。

### 11.5 /stop 的收尾清理

```
1. 取消 asyncio 流式任务
2. 清除 checkpoint 中残留的 interrupt_type + pending_* 字段
   （防止前端收到 running=false 后因 interrupt_type 非空而重新弹出审批面板）
3. 修复未应答的 tool_calls：
   for each missing tool_call_id:
       写入 ToolMessage({"cancelled": True, "reason": "用户停止了操作"})
   （保证下次 send_message 时 LLM 不会因 history 格式残缺而 400）
4. 补打本轮新消息的时间戳
5. 推送 running=false 的最终快照
```

### 12.2 状态同步的完整流程

```
用户发消息 → sendMessage()
  │
  ├── 乐观追加用户消息到 messages
  ├── busy = true
  ├── connect() → 开启 SSE（若未连接）
  ├── POST /threads/{id}/messages
  │   └── 后端 202 → 启动 _start_stream → graph.astream()
  │
  └── SSE 事件到达 → applyState(data)
      ├── messages 全量替换
      ├── interrupt_type → interrupt (驱动审批面板)
      ├── pending_* → 审批面板数据
      ├── thinking → currentThinking (思考链实时显示)
      ├── radiomics_progress → radiomicsProgress (进度条)
      ├── subagent → subagentStatuses (子Agent 状态卡片)
      ├── todos → todos (右侧步骤面板)
      ├── running → busy
      └── context_usage → contextUsage (token 用量指示器)

流结束 → agent_end 事件
  ├── busy = false
  ├── radiomicsProgress = null
  ├── currentThinking = null
  ├── runningThreadIds.delete(threadId)
  ├── updateRunningPolling()
  └── syncThread() → 最终 state 同步（兜底收敛）
```

### 12.3 审批流程的前端视角

```
SSE 推送 interrupt_type = "file_plan"
  → interrupt = "file_plan"
  → pendingPlan = {...}
  → busy = true (running=true)
  → 渲染 PlanEditor.vue（文件操作计划表格）

用户编辑计划 → PUT /threads/{id}/plan
  → 写回 checkpoint 的 pending_plan
  → 前端重新渲染

用户点击 [确认]
  → POST /confirm
  → busy = true
  → 后端 Command(resume={"action": "confirm"})
  → 继续 SSE 推送流
  → execute_confirmed 执行 → clear interrupt → 继续 call_llm
  → 下一轮 SSE 推送 interrupt_type = null, pendingPlan = null
  → PlanEditor 关闭

用户点击 [取消]
  → POST /cancel
  → 同上，但 execute_confirmed 走取消分支
  → cancelled ToolMessage + 可选 HumanMessage

用户点击 [其他...]
  → POST /other {instruction: "..."}
  → 同上，但带 other_instruction
```

### 12.4 轮询与运行状态收敛

```
问题: runningThreadIds 是前端内存中的状态，刷新后会丢失。
     侧边栏如何知道哪些对话正在运行？

方案: 双重来源
  1. 前端主动标记: sendMessage / confirm / cancel 后立即
     runningThreadIds.add(threadId)
  2. 后端列表响应: listThreads() 响应中每个 thread 带 running 标志
     syncRunningFromList() 合并进 runningThreadIds

     刚结束的对话: wasRunning=true 且 nowRunning=false
     → runningThreadIds.delete(id)
     → 如果用户正在看 → 不加 finishedThreadIds（已实时看到结束）
     → 否则 → finishedThreadIds.add(id)（侧边栏显示提示点）

轮询: 当 runningThreadIds 非空时，每 3 秒 pollRunningProjects()
      刷新所有含运行中对话的项目的列表
      运行中集合清空后自动停止轮询
```

## 13. 数据库设计

### 13.1 SQLite 表结构

```sql
-- 项目与对话在同一个 SQLite 文件（~/.onerad/projects.db 或自定义路径）
-- 由 ProjectStore 管理

CREATE TABLE projects (
    id TEXT PRIMARY KEY,          -- UUID
    name TEXT NOT NULL UNIQUE,
    path TEXT NOT NULL UNIQUE,    -- 项目根目录绝对路径
    description TEXT,
    created_at TEXT NOT NULL,     -- ISO 8601
    updated_at TEXT NOT NULL
);

CREATE TABLE threads (
    id TEXT PRIMARY KEY,          -- UUID (即 LangGraph thread_id)
    project_id TEXT NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    title TEXT NOT NULL DEFAULT '',
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);

-- SSE 事件持久化
CREATE TABLE sse_events (
    scope TEXT NOT NULL,          -- "agent" | "pipeline"
    scope_id TEXT NOT NULL,       -- thread_id
    event_id INTEGER NOT NULL,
    data TEXT NOT NULL,           -- JSON 字符串
    created_at TEXT NOT NULL,
    PRIMARY KEY (scope, scope_id, event_id)
);

-- LangGraph checkpoint (由 AsyncSqliteSaver 管理)
-- 内部表: checkpoint, checkpoint_blobs, checkpoint_writes
```

### 13.2 checkpoint 与 thread 的关系

- **thread 表**（ProjectStore）：存储对话元数据（标题、项目归属、时间戳）
- **checkpoint 表**（AsyncSqliteSaver）：存储 LangGraph 的完整状态快照（messages、state 字段）
- **sse_events 表**（ProjectStore）：存储 SSE 事件用于重连回放

三者通过 `thread_id` 关联。删除线程时需同时清理三处（见 `delete_thread` 端点）。

## 14. 关键设计决策与权衡

| 决策 | 选择 | 替代方案 | 理由与权衡 |
|------|------|---------|-----------|
| Agent 框架 | **LangGraph StateGraph** | 自建状态机 / LangChain AgentExecutor | `interrupt()` 原生支持人机协同；`AsyncSqliteSaver` 提供生产级 checkpoint；状态图显式建模比 AgentExecutor 的隐式循环更可控 |
| LLM 调用方式 | **原生 OpenAI SDK** | LangChain ChatOpenAI | DeepSeek 的 `reasoning_content` 是 OpenAI 协议的非标准扩展。LangChain 的流式拼接逻辑不认识此字段，直接丢弃。原生 SDK 保留完整控制 |
| LLM 模型 | **DeepSeek V4** | OpenAI GPT-4o / Claude | 成本（DeepSeek 便宜 10-20x）、中文推理能力、`reasoning_content` 思考链可见性 |
| API Key 存储 | **RunnableConfig 内存** | 写入 AgentState | 防止密钥随 checkpoint 持久化到 SQLite。删除线程时同步从内存清除 |
| Skill 加载 | **每次从磁盘读取** | 启动时缓存 / 写入代码常量 | 热更新——调整 prompt 后立即生效无需重启。磁盘 IO 开销可忽略（每个 LLM 调用读一次 ~3KB 文件 vs 数秒的 API 调用） |
| 子Agent 嵌套深度 | **限制为 1** | 无限制 / 递归 | 一层分派已覆盖主场景（项目探索 + 复杂计算子任务）；更深嵌套增加调试难度且收益递减 |
| 并行子Agent 并发 | **最多 4 worker** | 不限制 | 平衡 DeepSeek API 速率限制（同一 key 的并发请求）、本地内存占用、实际加速比（项目探索场景 3-4 个任务最优） |
| 子Agent 步数上限 | **150 superstep (~37 轮工具调用)** | 默认 25 | 子Agent 的整个生命周期在单次 stream 内完成，无分段预算；37 轮足以完成复杂的数据探索和分析任务 |
| 子Agent 结果截断 | **4000 字符** | 不截断 / 更短 | 4000 字符足以传达结论、文件路径和关键数字；过短丢失信息，过长消耗主对话上下文 |
| 同一轮 tool_calls 限制 | **最多 1 个需确认** | 允许多个 | 防止 `pending_*` 互相覆盖（状态字段非列表，只有一份）。多个需确认工具 LLM 可串行分轮发送 |
| high 风险脚本 | **标记高危 + 用户知情确认后执行** | 直接拒绝 | 把决策权交回用户。合法任务（如 `requests` 下载数据集）不应被技术手段阻止 |
| 中危脚本沙箱 | **注入沙箱头 wrapper** | 静态拒绝 / Docker | 运行时 hook `open()` 是 best-effort 防御，不是安全边界。在 windows 上比 Docker 更轻量且无额外依赖 |
| SSE 高频事件 | **不持久化** | 全部持久化 | 避免 SQLite 写入成为瓶颈（thinking 每秒可产数十 chunk）。重连靠 values 快照兜底（完整思考链在 AIMessage.additional_kwargs 中） |
| `parallel_tool_calls` | **关闭 (False)** | 开启 | 杜绝单轮内多个需确认 tool_calls 的覆盖问题；也降低单轮 token 峰值和后续处理复杂度 |
| 文件操作备份 | **执行前自动备份到 `.onerad_backup/`** | 不做备份 | move/copy 的覆盖操作不可逆；时间戳子目录实现简单快照 |
| 特征提取断点续提 | **基于 h5 缓存 + YAML hash 的断点续提** | 每次全量重跑 | 30 个病例提取可能需 10-30 分钟；中途失败不应从头开始 |
| 前端 busy 状态 | **完全由后端 `running` 字段驱动** | 由 `interrupt_type` 推断 | `execute_confirmed` 清除 pending 前的中间快照仍带旧 interrupt_type，用其推断 busy 会导致 UI 闪烁 |
| 前端乐观更新 | **用户消息乐观追加到 messages，失败回滚** | 等后端确认后再显示 | 消息发送延迟需数百 ms；乐观更新消除感知延迟。冲突由后端的 409（流忙碌/待确认）兜底 |

### 16.3 向后兼容约定

- `AgentState.api_key` 标记为 `NotRequired`：新线程不再写入此字段，但旧 checkpoint 可能仍有值。`_resolve_api_key` 的优先级：`configurable` → `state` → 环境变量
- `AgentState.model` 写入受支持模型名；`_resolve_model` 对不在 `DEEPSEEK_MODELS` 中的值回退到默认模型（`deepseek-v4-flash`）
- 取消操作时的 `HumanMessage` 追加只针对重操作类型（`file_plan`/`python_script`/`radiomics_*`/`feature_statistics`/`subagent_dispatch`），简单查询不追加
- `executed: True` 注入：对 dict 结果做键注入保留原结构，对非 dict 结果包裹为 `{"executed": True, "results": ...}`
- `/stop` 的 checkpoint 清理：清除残留 `interrupt_type` 后，若 snapshot 仍含未应答 tool_calls，为每个缺失 id 补 ToolMessage（保证后续 LLM 调用不 400）
