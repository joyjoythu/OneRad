# 数据准备

OneRad 对数据组织方式有少量约定，准备得当可以让 Agent 一次跑通全流程。

## 推荐目录结构

```
项目目录/
├── images/              # 影像文件（.nii.gz）
│   ├── patient001.nii.gz
│   ├── patient002.nii.gz
│   └── ...
├── masks/               # 分割掩膜（.nii.gz）
│   ├── patient001_mask.nii.gz
│   ├── patient002_mask.nii.gz
│   └── ...
└── clinical.xlsx        # 临床表格（.xlsx / .csv 均可）
```

::: tip 目录名不是硬要求
Agent 会递归扫描项目目录，即使你的目录结构不同，它也能通过探索搞清楚数据在哪。但使用 `images/` + `masks/` 的约定结构，配对发现会更直接、置信度更高。
:::

## 影像与掩膜的命名

配对发现按以下优先级自动匹配：

1. **文件名后缀匹配** — 掩膜文件名以 `_mask` / `_seg` / `_label` / `_roi` 结尾（如 `patient001_mask.nii.gz`）
2. **token 交集匹配** — 文件名分词后与影像文件名重合度最高
3. **患者 ID 前缀匹配** — 按患者编号前缀关联

配对结果按置信度分为**高 / 中 / 低**三级展示，未匹配的影像和掩膜单独列出。确认前请重点检查中、低置信度的配对。

## DICOM 转 NIfTI

临床拿到的原始影像通常是 DICOM 序列，影像组学分析需要 NIfTI 格式。不需要自己写脚本，直接对 Agent 说：

> **帮我把这些 DICOM 转成 nii.gz**

Agent 的处理逻辑：

- **任意嵌套结构支持**：递归扫描输入目录，任何含 `.dcm` 文件的目录视为一个转换单元，通过 GDCM 自动识别其中的 DICOM 序列（按 SeriesInstanceUID 分组）
- **同目录多序列**：同一目录下存在多个序列（如 T1 + T2）时各自输出独立文件，通过序列描述（SeriesDescription）区分；描述缺失时回退为 `series{N}`
- **命名规则**：`<目录名>_<序列描述>.nii.gz`，重名自动追加 `_2`、`_3` 后缀，特殊字符替换为下划线
- **输出镜像输入结构**：嵌套目录的相对路径保留到输出侧
- **进度可见**：转换过程有逐序列进度条；单个序列失败只记录错误，不中断其余序列

转换前会弹出确认面板，展示输入/输出目录和扫描到的序列清单（含每序列描述与切片数），确认后执行。

## 临床表格要求

| 要求 | 说明 |
|------|------|
| **患者 ID 列** | 必须有一列能把每行对应到影像文件（如 `PATIENT_ID`）。Agent 会自动识别，识别不了会向你提问 |
| **Label 列** | 二分类结局变量，取值 0/1（如 `1 恶性 0 良性`） |
| **协变量** | 可选，临床特征列（年龄、分期等），建模时作为候选变量 |
| **中文列名** | 支持。Agent 会自动把中文列名翻译成英文，避免 SHAP 图乱码 |

::: tip
列名确认过一次后会被记入[项目记忆](/features/project-memory)，之后新对话无需重复确认。
:::

## PyRadiomics 参数配置（YAML）

特征提取参数由项目内的 YAML 文件管理。Agent 会自动检查图像实测 spacing 与 YAML 配置的 `resampledPixelSpacing` 是否一致，不一致时建议调整。

常用参数说明见 [PyRadiomics 参数配置](/reference/yaml-config)。你也可以直接对 Agent 说「**把 binWidth 改成 10**」，它会用保留注释和格式的方式修改 YAML。

## 数据检查清单

开始分析前，快速过一遍：

1. ✅ 影像和掩膜都是 `.nii.gz`（或已安排 DICOM 转换）
2. ✅ 影像与掩膜文件名可对应（同前缀 + 后缀约定）
3. ✅ 临床表有患者 ID 列和 0/1 Label 列
4. ✅ 数据已放入项目目录（或挂载的数据根目录）

准备就绪 → [5 分钟快速上手](/guide/quickstart)
